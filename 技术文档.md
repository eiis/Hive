# Hive — 多 Agent 协作框架技术文档

> 开源多模型 Agent 协作平台，支持任务自动分解、实时状态可视化、MCP 集成。

---

## 一、项目概述

### 1.1 目标

构建一个类似 CCCC 的多 Agent 协作系统，并在以下方面超越它：

- **实时工作状态可视化**：群聊中每个 Agent 的状态、当前任务、进度一目了然
- **多模型支持**：Claude、GPT-4、Gemini 等模型可混合使用
- **MCP 集成**：可作为 MCP Server 嵌入 Claude Code 工作流
- **开源友好**：Docker 一键部署，插件化模型接入

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **Agent** | 由 AI 模型驱动的工作单元，有独立状态和任务队列 |
| **Foreman** | 特殊 Agent，负责理解用户意图、分解任务、分配给其他 Agent |
| **Group** | Agent 的协作空间，共享上下文和消息 |
| **Event Bus** | 所有状态变化和消息通过统一事件总线流转 |
| **Shared Context** | Agent 间共享的目标、记忆、任务树 |

## 二、技术栈

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 后端框架 | Python + FastAPI | AI 生态成熟，asyncio 支持并发 Agent |
| 实时通信 | WebSocket（FastAPI 内置） | 低延迟推送状态和消息 |
| 数据库 | SQLite → PostgreSQL | 开发简单，生产可无缝迁移 |
| ORM | SQLAlchemy + Alembic | 迁移管理方便 |
| 前端框架 | React + TypeScript + Vite | 类型安全，生态丰富 |
| 前端 UI | Tailwind CSS + shadcn/ui | 快速构建现代界面 |
| MCP | 官方 Python MCP SDK | 标准协议集成 Claude Code |
| 包管理 | uv（后端）+ pnpm（前端） | 速度快 |
| 部署 | Docker + Docker Compose | 一键启动 |

## 四、核心模块设计

### 4.1 Model Adapter（模型适配层）

所有模型实现统一接口，业务层无需关心底层模型：

```python
class BaseModelAdapter(ABC):
    @abstractmethod
    async def chat(
        self,
        messages: list[Message],
        tools: list[Tool] | None = None,
        stream: bool = True,
    ) -> AsyncIterator[str]: ...

class ClaudeAdapter(BaseModelAdapter): ...
class OpenAIAdapter(BaseModelAdapter): ...
class GeminiAdapter(BaseModelAdapter): ...
```

支持任何 OpenAI 兼容接口（Ollama、DeepSeek、Qwen 等）通过 `OpenAIAdapter` 接入。

### 4.2 Agent 状态机

```
         用户/Foreman 分配任务
              │
         ┌────▼────┐
    ┌────│  idle   │────┐
    │    └─────────┘    │
    │         │ 收到任务  │
    │    ┌────▼────┐    │
    │    │thinking │    │  ← 正在理解任务，制定计划
    │    └────┬────┘    │
    │         │         │
    │    ┌────▼────┐    │
    │    │ working │    │  ← 调用工具、生成内容
    │    └────┬────┘    │
    │    ┌────┴────┐    │
    │    │         │    │
    │ ┌──▼──┐  ┌───▼──┐ │
    │ │done │  │failed│ │
    │ └──┬──┘  └───┬──┘ │
    └────┴──────────┘   │
              │ 状态重置  │
              └──────────┘

    waiting：等待其他 Agent 完成依赖任务
```

每次状态变化都通过 Event Bus 广播，前端实时更新面板。

### 4.3 Event Bus

```python
# 事件类型
class EventType(str, Enum):
    AGENT_STATUS_CHANGED  = "agent.status.changed"
    AGENT_PROGRESS        = "agent.progress"
    MESSAGE_SENT          = "message.sent"
    TASK_CREATED          = "task.created"
    TASK_UPDATED          = "task.updated"
    TASK_COMPLETED        = "task.completed"

# Agent 工作时上报
await event_bus.emit(EventType.AGENT_STATUS_CHANGED, {
    "agent_id": "claude-1",
    "state": "working",
    "detail": "正在写 TodoList.tsx",
    "progress": 0.45,
})
```

WebSocket Handler 订阅所有事件，推送到前端。

### 4.4 Message Router

支持多种寻址方式：

| 地址 | 含义 |
|------|------|
| `@all` | 广播给 Group 内所有成员 |
| `@foreman` | 发给协调者 |
| `@peers` | 发给除自己外的所有 Agent |
| `@user` | 发给用户 |
| `@{name}` | 发给指定 Agent |

### 4.5 Foreman 设计

Foreman 是一个特殊 Agent，具备：

1. **意图理解**：分析用户需求
2. **任务分解**：将需求拆成有依赖关系的子任务树
3. **智能分配**：根据 Agent 当前状态和能力分配任务
4. **进度汇总**：收集各 Agent 结果，整合回复用户

```
用户："写一个 React Todo 应用，有增删改查功能"

Foreman 分解结果：
Todo App
├── T1: 初始化项目结构          → claude  (无依赖)
├── T2: 实现数据层 types.ts     → claude  (依赖 T1)
├── T3: 实现 TodoItem 组件      → claude  (依赖 T2)
├── T4: 实现 TodoList 组件      → claude  (依赖 T3)
├── T5: 实现 App.tsx 主页面     → claude  (依赖 T4)
└── T6: 写单元测试              → codex   (依赖 T2)
```

### 4.6 Shared Context

Agent 间共享的数据结构：

```python
class SharedContext(BaseModel):
    objective: str           # 本次协作的总目标
    task_tree: list[Task]    # 任务树（含依赖关系）
    memory: list[MemoryItem] # 共享记忆（关键结论/决策）
    artifacts: list[Artifact]# 产出物（文件、代码片段等）
```

### 4.7 MCP Server

暴露以下工具供 Claude Code 调用：

| 工具名 | 功能 |
|--------|------|
| `hive_send_message` | 向 Group 发消息 |
| `hive_get_task_status` | 查询任务状态 |
| `hive_get_context` | 读取共享上下文 |
| `hive_list_agents` | 列出所有 Agent 及其状态 |
| `hive_assign_task` | 直接给指定 Agent 分配任务 |

## 八、关键设计决策

### 8.1 为什么用 Event Bus 而不是直接 WebSocket 推送？

Event Bus 作为内部消息中枢，WebSocket 只是其中一个订阅者。这样：
- 后端逻辑不依赖前端连接
- 方便日志记录和事件回放
- 未来可以换成 Redis Pub/Sub 而不改业务代码

### 8.2 Foreman 如何保证任务分解质量？

1. System Prompt 明确定义 Foreman 的角色和输出格式（JSON 任务树）
2. 结构化输出（Claude 支持 `response_format`）确保格式正确
3. 任务分配时考虑 Agent 当前负载和能力声明

### 8.3 Agent 进度如何计算？

- **确定性进度**：有明确步骤时（如读 N 个文件），按步骤计算
- **估算进度**：无明确步骤时，由 Agent 自行在 System Prompt 中被要求定期上报 0-1 的进度值
- **前端平滑处理**：避免进度跳跃，使用动画过渡
